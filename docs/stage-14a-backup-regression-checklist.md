# 备份与恢复 Regression Checklist

这份清单用于覆盖第十四A到第十五阶段的备份、列表读取、严格校验、恢复诊断与相关保护，不替代现有主 smoke checklist。

## 1. 手动备份：空资源

步骤：
1. 启动应用，确保当前数据目录下没有正文图片和 notebook 封面。
2. 进入“设置”，点击“立即创建备份”。

预期：
1. 能正常创建备份，不报错。
2. 新备份项会立即出现在列表顶部。
3. 后台补刷新完成后，列表无重复项，顺序稳定。
4. 控制台能看到 `[backup.perf]` 分阶段日志。

## 2. 手动备份：小资源

步骤：
1. 在某个 note 中插入 1 张小图片，并为当前 notebook 设置 1 张封面。
2. 进入“设置”，点击“立即创建备份”。

预期：
1. 创建成功提示正常出现。
2. 新备份无需等待全量重扫即可立即显示。
3. 后台补刷新后，该备份仍只出现 1 条。

## 3. 手动备份：多资源

步骤：
1. 准备多个包含正文图片和封面的 notebook / note。
2. 进入“设置”，点击“立即创建备份”。

预期：
1. 备份能够成功完成。
2. 列表立即插入新项。
3. 后台刷新后无重复、无顺序错乱。
4. 控制台日志能看到数据库快照、资源打包、zip 完成、结果构造等阶段。

## 4. 图片 note + notebook 封面 + 备份恢复 roundtrip

步骤：
1. 在某个 note 中插入正文图片，并为对应 notebook 设置封面。
2. 创建一份新备份。
3. 修改当前数据，例如删除图片内容或清除封面。
4. 恢复刚才创建的备份。

预期：
1. restore 前会继续走严格校验。
2. reload 后，正文图片和 notebook 封面都回到备份时状态。
3. 资源路径不失效，图片和封面都能正常显示。

## 5. 从 notebooks 切到 settings 后再手动备份

步骤：
1. 在 notebooks 中打开一个 note，输入一段新内容。
2. 不等待自动保存，直接切到“设置”。
3. 到达“设置”后立刻点击“立即创建备份”。

预期：
1. 切换 section 前会先走 flush 保护，不会静默丢稿。
2. 进入设置后可正常创建备份。
3. 备份列表仍按“立即插入 + 后台补刷新”工作。

## 6. 恢复备份前 dirty note 阻断

步骤：
1. 打开一个 note，制造未保存内容。
2. 切到“设置”，选择某份备份并点击恢复。

预期：
1. 恢复前仍会调用 notebooks flush guard。
2. 若保存失败，应阻止 restore，并显示明确中文提示。
3. 备份优化不应绕过这条保护链路。

## 7. 设置页首次打开时备份列表正常

步骤：
1. 准备多份现有备份。
2. 打开应用并进入“设置”。

预期：
1. 首次打开设置页时仍会从真实目录读取备份列表。
2. 列表内容、排序与可恢复状态正常。
3. 控制台能看到 `reason=initial-load` 的刷新 perf 日志。

## 8. 创建备份后列表立即可见且最终刷新正确

步骤：
1. 保持“设置”页打开。
2. 连续创建 1 到 2 次备份，观察列表变化。

预期：
1. 每次创建成功后，新项都会立即出现在列表顶部。
2. 后台刷新完成后，不应出现重复项、顺序闪烁或旧请求覆盖新结果。
3. 控制台能看到 `reason=post-create` 的刷新 perf 日志。

## 9. 设置页首开先显示轻列表

步骤：
1. 准备多份备份，其中至少包含 1 份损坏 zip。
2. 打开应用并进入“设置”。

预期：
1. 设置页主体应先渲染完成，备份列表优先显示文件名、时间、大小等基础信息。
2. 首次进入设置页时，不会等待全部备份严格校验完成后才显示列表。
3. 损坏 zip 在首屏仍先显示为 `待校验`，不会因为列表读取直接报错。

## 10. 首屏前 5 项后台补校验

步骤：
1. 保持“设置”页打开，观察列表状态变化。
2. 若备份数超过 5 份，重点观察排序靠前的前 5 项。

预期：
1. 前 5 项会按串行方式从 `待校验` 变为 `校验中`，再变为 `可恢复` 或 `损坏不可恢复`。
2. 其余项保持 `待校验`，直到用户手动校验或准备恢复时再触发严格校验。
3. 页面不应因后台补校验而明显卡顿。

## 11. 单项显式校验与恢复前强制校验

步骤：
1. 对一个 `待校验` 项点击“校验备份”。
2. 对另一个 `待校验` 项直接点击“校验后恢复”。

预期：
1. 两种入口都只会校验当前单项备份，不会触发全量扫描。
2. 校验通过后，该项状态变为 `可恢复`。
3. 校验失败后，该项状态变为 `损坏不可恢复`，并显示明确中文原因。

## 12. 非法 zip 被拒绝

步骤：
1. 在备份目录中放入一个扩展名为 `.zip`、但内容不是合法 zip 的文件。
2. 进入“设置”，等待该项被后台补校验，或手动点击“校验备份”。

预期：
1. 该项应被标记为 `损坏不可恢复`。
2. 错误原因应明确指向“备份压缩包无法读取或已损坏”。

## 13. 缺 manifest / schemaVersion 不兼容被拒绝

步骤：
1. 分别准备一个缺少 `manifest.json` 的 zip 和一个 `schemaVersion` 高于当前应用支持版本的 zip。
2. 在设置页对这两份备份执行严格校验。

预期：
1. 缺 manifest 的备份会被拒绝，并显示清单损坏类中文提示。
2. `schemaVersion` 不兼容的备份会被拒绝，并显示版本不兼容类中文提示。

## 14. restore 成功后的搜索 / 标签 / 复习 / 图片 / 封面 roundtrip

步骤：
1. 准备一份合法备份，内容包含：
   - note 正文图片
   - notebook cover
   - 可搜索正文
   - 标签关联
   - 复习方案与任务
2. 在当前数据上做明显修改后，恢复这份备份。

预期：
1. 应用恢复成功并自动 reload。
2. reload 后搜索仍可命中原 note。
3. 标签系统仍可打开关联 note。
4. 复习日历中的任务仍可查看并点击打开 note。
5. 正文图片与 notebook cover 都能正常显示。

## 15. restore 失败诊断与 dirty note 阻断

步骤：
1. 在 notebooks 中制造未保存内容，直接去设置页尝试恢复备份。
2. 再分别验证非法 zip、缺 manifest、settings 损坏、资源目录异常等失败场景。

预期：
1. dirty note 时，restore 前仍会先走现有 flush guard；若保存失败，应阻止恢复。
2. restore 失败时，前端应显示明确中文原因，而不是统一“恢复失败”。
3. Rust 控制台应输出对应阶段的 `[backup.perf]` start / complete / failed 日志。
